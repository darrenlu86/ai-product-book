# 第 1 章｜AI 改變了什麼，沒改變什麼

我是一個不會寫程式的產品經理。

這句話在兩年前說出來，大概會換來一個禮貌性的微笑，然後對方心裡想：「那你做產品的時候，工程師不會很想打你嗎？」老實說，有時候確實會。

但現在我可以接著說：「不過我用 AI 做了十六個 Side Project。」

對方的表情通常會從禮貌變成困惑。十六個？什麼意思？你不是說你不會寫程式？

沒錯，我不會寫程式。我在一間台灣的網路認證公司當產品創新經理，日常工作是寫規格、畫流程、跟工程師開會。我的程式能力大概停留在大學時期用 Python 寫過幾個作業的程度，那些作業我還記得交出去的時候有一半是跑不動的。

可是在過去這段時間，我做了婚禮邀請函網站、考題庫平台、AI 口述歷史工具、桌面聽寫 App、自動化影片管線、婚禮感謝卡 SaaS......這些東西不是玩具，有些真的有人在用，有些甚至有在收費。

這件事到底是怎麼發生的？


## 改變了的事

最明顯的改變是：**寫程式的門檻被炸掉了**。

以前你想做一個網站，得先學 HTML、CSS、JavaScript，然後學一個前端框架（React 或 Vue），然後學後端語言（Python 或 Node.js），然後學資料庫（PostgreSQL 或 MongoDB），然後學怎麼部署到伺服器上......光是列出這些名詞就夠嚇人了，更不用說每一個都有自己的生態系和一堆奇怪的設定檔。

現在呢？我打開終端機，用自然語言跟 AI 說：「幫我建一個婚禮邀請函網站，要有倒數計時、Google Maps 地圖、RSVP 表單。」它就開始生成程式碼了。

我第一次做這件事的時候，坐在電腦前看著程式碼一行一行冒出來，心裡的感覺很奇妙。不是興奮，比較像是：「等一下，這合法嗎？」就好像你一直以為要拿到建築師執照才能蓋房子，結果突然有個人跟你說：「你只要跟我說你想住什麼樣的房子，我來蓋。」

第二個改變是：**一個人可以當一個團隊**。

傳統的軟體開發，你至少需要前端工程師、後端工程師、設計師，可能還要一個 DevOps（負責伺服器維運的工程師）。一個最小可行的團隊大概要三到五個人。現在一個人配上 AI，可以同時處理前端介面、後端邏輯、資料庫設計、伺服器部署。

我做那十六個專案的時候，沒有任何一個有其他工程師參與。全部都是我一個人加上 AI。

這聽起來很厲害對吧？

先別急著崇拜，因為接下來要講的才是重點。


## 沒改變的事

AI 沒有改變的第一件事：**你還是得知道軟體是由哪些部分組成的**。

我做婚禮邀請函網站的時候，一開始很順利。AI 幫我生成了漂亮的頁面，有倒數計時、有地圖、有動畫效果。我超開心，覺得自己是天才。

然後我想加一個 RSVP 功能，讓賓客可以線上回覆出不出席。AI 也幫我寫好了表單，但我按下送出之後......什麼都沒發生。

為什麼？因為表單需要一個後端伺服器來接收資料，還需要一個資料庫來存資料。而我只有前端頁面。

那一刻我才意識到，我根本不知道一個「可以接收表單資料」的網站需要哪些東西。AI 可以幫我寫任何一個部分的程式碼，但如果我不知道整體架構長什麼樣子，我連問題都問不出來。

這就像你去餐廳，跟服務生說「我要吃飯」，但你不知道菜單上有什麼。服務生可以幫你上任何一道菜，但你得先知道有前菜、主菜、甜點這些分類，才能做出有意義的選擇。

AI 沒有改變的第二件事：你還是得能判斷 AI 做得對不對。

AI 不是神。它會犯錯，而且犯的錯經常看起來很有道理。它會信心滿滿地告訴你一段程式碼可以解決你的問題，然後那段程式碼跑下去直接把你的資料庫清空。（別問我怎麼知道的。）

如果你完全不懂軟體的基本概念，你就沒辦法做品質把關。你會變成一個只能按「確認」的橡皮圖章，而不是一個真正在指揮開發的人。


## 關鍵翻轉：從 HOW 到 WHAT 和 WHY

我們來看看新舊時代的核心差異：

```
 ┌─────────────────────────────────────────────────┐
 │              舊時代（AI 以前）                     │
 │                                                   │
 │   稀缺的是 HOW                                    │
 │   ─────────────                                   │
 │   • 怎麼寫一個 API？                               │
 │   • 怎麼設定資料庫連線？                            │
 │   • 怎麼處理使用者認證？                            │
 │                                                   │
 │   你需要：花幾年學技術                              │
 │   瓶頸：找不到會寫的人                              │
 └─────────────────────────────────────────────────┘

                      │
                      ▼

 ┌─────────────────────────────────────────────────┐
 │              新時代（AI 以後）                     │
 │                                                   │
 │   稀缺的是 WHAT 和 WHY                            │
 │   ─────────────────────                           │
 │   • 要解決什麼問題？                               │
 │   • 為什麼使用者需要這個功能？                      │
 │   • 什麼才是正確的產品決策？                        │
 │                                                   │
 │   你需要：理解問題 + 判斷能力                       │
 │   瓶頸：不知道要做什麼                              │
 └─────────────────────────────────────────────────┘
```

以前的世界，技術能力是最值錢的東西。你會寫程式，你就是稀缺人才。公司花大錢請資深工程師，因為他們知道 HOW——怎麼把東西做出來。

現在 AI 可以搞定大部分的 HOW。它寫程式的速度比人快十倍，而且不用休息、不會請假、不會因為需求改了第七次就暴怒。

但 AI 不知道 WHAT——你到底要做什麼。它也不知道 WHY——為什麼要做這個而不是做那個。

這些判斷，是人的工作。是你的工作。

我做婚禮邀請函的時候，AI 不會告訴我：「欸你應該把 RSVP 的截止日期設在婚禮前兩週，因為你需要提前跟餐廳確認桌數。」這是我自己知道的事情，因為我正在籌備自己的婚禮。

AI 也不會告訴我：「你的邀請函不需要做多國語言，因為你的賓客都是台灣人。」這種產品判斷，需要的是對使用者的理解，不是技術能力。

所以如果你正在翻這本書，心裡想著「我不會寫程式，所以我沒辦法做產品」，我要跟你說：你搞反了。

在 AI 時代，知道要做什麼和為什麼要做，比知道怎麼做更重要。而前者從來就不是工程師的專利。


## 那我需要學什麼？

你不需要變成工程師。你需要的是一張地圖。

蓋房子不需要會砌磚，但你得知道房子有地基、有牆壁、有屋頂、有水電管線。你不需要自己去挖地基，但你得知道沒有地基的房子會倒。

軟體也一樣。你不需要會寫每一行程式碼，但你得知道一個軟體產品是由哪些部分組成的，它們之間怎麼互動，哪些地方容易出問題。

這就是這本書要做的事。

下一章，我們來看那張地圖。

# 第 2 章｜指揮 AI 的前提：你腦中要有地圖

我在做 ExamBank（考題庫平台）的時候，被一個叫做 CORS 的東西卡了整整一天。

那天的狀況是這樣的：前端網站做好了，後端 API 也做好了，兩邊分開測試都沒問題。但是當我在瀏覽器上打開網站，讓它去呼叫後端 API 的時候，瀏覽器噴了一個紅色的錯誤訊息，大概長這樣：

```
Access to fetch at 'https://api.exambank.com' from origin
'https://exambank.com' has been blocked by CORS policy
```

我看著這段訊息，心裡想：CORS 是什麼鬼？

我問 AI，它跟我解釋了一大堆：Cross-Origin Resource Sharing，跨來源資源共享，是瀏覽器的安全機制，當你的前端和後端在不同的網域（domain）上時，瀏覽器會擋住請求，除非後端明確允許......

我聽完之後的反應是：「所以你的意思是，我的前端跟後端都沒有 bug，但它們就是不能溝通？因為瀏覽器不讓它們？」

對。就是這麼荒謬。

但這件事的重點不在 CORS 本身。重點是：**如果我事先知道前端和後端是分開的兩個東西，而且它們之間的溝通有一些規則要遵守，我可能根本不會被卡住。**

我會在一開始就問 AI：「我的前端和後端是不同網域的，有什麼要注意的嗎？」然後 AI 會告訴我要設定 CORS，五分鐘搞定。

但因為我腦中沒有那張「軟體架構地圖」，我連問題都問不出來，只能看著紅色錯誤訊息發呆一整天。

這就是為什麼你需要那張地圖。


## 軟體產品的分層架構

一個現代的軟體產品，大致上長這樣：

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   使用者（User）                                         │
│   就是你、我、用手機或電腦操作的人                         │
│                                                         │
└─────────────────────┬───────────────────────────────────┘
                      │ 點擊、輸入、滑動
                      ▼
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   前端（Frontend）                                       │
│   你看得到的部分：按鈕、文字、圖片、表單                   │
│   技術：HTML、CSS、JavaScript、React、Next.js             │
│                                                         │
└─────────────────────┬───────────────────────────────────┘
                      │ 透過 API 傳送/接收資料
                      ▼
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   後端（Backend）                                        │
│   你看不到的部分：處理邏輯、驗證身分、運算                 │
│   技術：Python、Node.js、FastAPI、Express                 │
│                                                         │
└─────────────────────┬───────────────────────────────────┘
                      │ 讀取/寫入資料
                      ▼
┌─────────────────────────────────────────────────────────┐
│                                                         │
│   資料庫（Database）                                     │
│   存放所有資料的地方：會員帳號、訂單、文章                 │
│   技術：PostgreSQL、MySQL、Supabase                      │
│                                                         │
└─────────────────────────────────────────────────────────┘

         ↕ 整個東西跑在 ↕

┌─────────────────────────────────────────────────────────┐
│                                                         │
│   部署平台（Hosting / Deployment）                       │
│   把你的軟體放到網路上讓別人用                             │
│   技術：Cloudflare、Google Cloud、Vercel                  │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

看起來很多東西？別緊張。我用蓋房子來解釋一次。


## 蓋房子的比喻

```
軟體                          蓋房子
──────────────────────────────────────────────
前端 (Frontend)        ──→    室內裝潢、門窗
                              看得到、摸得到的部分

後端 (Backend)         ──→    水電管線、結構工程
                              看不到但讓一切運作的部分

資料庫 (Database)      ──→    倉庫 / 儲藏室
                              存放所有東西的地方

API                    ──→    管線接口
                              讓水電(資料)在不同
                              房間(系統)之間流通

部署 (Deployment)      ──→    蓋在哪塊地上
                              地段決定了租金和交通
```

你不需要會接水管，但你要知道房子裡有水管這件事。不然等到廁所馬桶不通的時候，你可能會以為是地板壞了。

軟體開發也一樣。你不需要會寫 API，但你要知道前端和後端之間是透過 API 在溝通的。不然等到出問題的時候（就像我的 CORS 故事），你連問題出在哪一層都搞不清楚。


## 每一層的白話翻譯

讓我再展開說明每一層，用最白話的方式。

**前端（Frontend）** 是使用者直接互動的那一層。你在手機上看到的每一個按鈕、每一行文字、每一張圖片，都是前端的工作。它就像餐廳的用餐區——客人看到的、坐到的、吃到的都在這裡。

後端（Backend）是在幕後處理邏輯的那一層。當你按下「送出訂單」，後端會驗證你的身分、計算價格、檢查庫存。它就像餐廳的廚房——客人看不到，但所有真正的工作都在這裡發生。

資料庫（Database）是存資料的地方。你的帳號密碼、你的訂單紀錄、你上傳的照片路徑，全部存在資料庫裡。它就像餐廳的食材倉庫——廚房需要什麼食材，就去倉庫拿。

**API（Application Programming Interface）** 是前端跟後端之間的溝通管道。前端不能直接去翻資料庫（就像客人不能直接衝進廚房），它得透過 API 這個「服務窗口」來點餐和取餐。我被 CORS 卡住的那次，問題就出在這個管道上。

部署（Deployment）是把你的軟體放到網路上的過程。你在自己電腦上做好的東西，別人是看不到的。你得把它「部署」到某個伺服器上，就像你在家做好了一桌菜，得端到餐廳去才有客人吃得到。


## 你不需要知道怎麼做，但要知道它們存在

這一點非常關鍵，我要再強調一次。

我做 ExamBank 的時候，整個架構是這樣的：

```
                    ┌──────────────┐
                    │   Cloudflare │
  使用者 ──────→    │   Pages      │  ← 前端部署在這
                    └──────┬───────┘
                           │ API 呼叫
                           ▼
                    ┌──────────────┐
                    │   Google     │
                    │   Cloud Run  │  ← 後端部署在這
                    └──────┬───────┘
                           │ 讀寫資料
                           ▼
                    ┌──────────────┐
                    │   Supabase   │  ← 資料庫在這
                    └──────────────┘
```

我不需要知道 Cloudflare Pages 底層是怎麼運作的。我不需要知道 Google Cloud Run 的容器技術細節。我不需要知道 Supabase 的 PostgreSQL 是怎麼做索引優化的。

但我需要知道：我的產品有三層，前端在 Cloudflare、後端在 Google Cloud、資料庫在 Supabase。它們是分開的，它們之間透過 API 溝通，而且這個溝通有一些規則（比如 CORS）。

有了這個認知，當出問題的時候，我至少能做到兩件事：

第一，我能縮小問題的範圍。「網站打不開」是前端的問題。「資料存不進去」可能是後端或資料庫的問題。「前端能開但資料顯示不出來」大概是 API 那一層的問題。

第二，我能問出正確的問題。與其跟 AI 說「我的網站壞了」（這等於跟醫生說「我不舒服」），我可以說「我的前端可以正常顯示，但呼叫後端 API 時收到錯誤回應，錯誤碼是 403」。這種精準的描述，會讓 AI 給你精準的答案。


## 其他你該知道的概念

除了那四層架構之外，還有幾個概念你遲早會碰到，我先簡單列在這裡：

**版本控制（Git）**——追蹤程式碼修改紀錄的系統。就像 Google Docs 的版本紀錄，讓你可以回到之前的版本。你不需要精通 Git，但你要知道它存在，而且它是避免災難的救命繩。

環境變數（Environment Variables）——存放密碼、API 金鑰等敏感資訊的地方。你不會把家裡的鑰匙直接貼在門上，同理你不該把密碼直接寫在程式碼裡。

第三方服務（Third-party Services）——別人做好的功能，你直接拿來用。寄 email 用 Resend、處理付款用 PayUNI、做使用者登入用 Supabase Auth。你不需要自己造輪子。

這些概念我們在後面的章節都會一一展開。現在你只需要知道它們存在，先把名字記住就好。


## 這本書就是那張地圖

回到最初的比喻。

你要指揮 AI 幫你蓋房子，你不需要自己會砌磚、接水管、裝電線。但你需要一張建築藍圖，上面標示了哪裡是地基、哪裡是承重牆、哪裡是水管、哪裡是電線。

沒有這張藍圖，你可能叫 AI 把承重牆拆掉來擴大客廳——然後整棟房子塌了。

沒有這張藍圖，你可能不知道為什麼馬桶不通——因為你不知道有水管這東西。

沒有這張藍圖，你可能蓋了一棟很漂亮的房子，但忘了留大門——因為你太專注在室內裝潢，忘了人要能走進來。

這本書要給你的，就是那張軟體產品的建築藍圖。

接下來的每一章，我們會一層一層地看過去。不講程式碼，只講概念。你讀完之後不會變成工程師，但你會變成一個知道怎麼指揮 AI 的人。

而在這個時代，那可能比會寫程式更有用。
